pub async fn ctrl_c() -> Result<()>
展开描述

当向进程发送âctrl-câ通知时完成。

虽然 Unix 和 Windows 处理信号的方式差异很大，但 两个平台都支持在âctrl-câ上接收信号。 此函数提供用于接收此通知的 可移植 API。

返回的 future 一旦被轮询，就会注册一个监听器。 此 future 将在首次接收到 ctrl-c 时完成， 该之后的初始调用为 Future::poll 或 .await。

§Caveats

在 Unix 平台上，第一次为 特定信号类型注册 Signal 实例时， 会安装一个操作系统信号处理程序， 该程序在接收到该信号时将替换 默认的平台行为，持续 整个进程期间。

例如，Unix 系统默认会在收到 终端上 "CTRL+C" 生成的信号时终止进程。但是，当创建 ctrl_c 流以监听此信号时，在其到达时， 它将被转换为流事件，并且进程将继续 执行。即使此 Signal 实例被丢弃， 后续的 SIGINT 传递最终也会被 Tokio 捕获，并且默认平台行为 将不会重置。

因此，应用程序应小心以确保 在监听特定信号后预期的信号行为 按预期发生。

§示例

use tokio::signal;

#[tokio::main]
async fn main() {
    println!("waiting for ctrl-c");

    signal::ctrl_c().await.expect("failed to listen for event");

    println!("received ctrl-c event");
}

在后台监听：

tokio::spawn(async move {
    tokio::signal::ctrl_c().await.unwrap();
    // Your handler here
});
