tokio::process

结构体 Command 

Source 
pub struct Command { /* private fields */ }
展开描述

此结构体模仿标准库中的 std::process::Command API，但将用于创建进程的函数替换为 异步变体。主要提供的 异步函数为 spawn、status 和 output。

Command 使用某些 std 类型的异步版本（例如 Child）。

pub fn new<S: AsRef<OsStr>>(program: S) -> Command

构造一个新的 Command，用于在 路径 program 启动程序，使用以下默认配置：

  • No arguments to the program
  • Inherit the current process’s environment
  • Inherit the current process’s working directory
  • Inherit stdin/stdout/stderr for spawn or status, but create pipes for output

提供了 Builder 方法来修改这些默认值并 otherwise configure the process.

如果 program 不是绝对路径，将以 操作系统定义的方式搜索 PATH。

要使用的搜索路径可以通过 在 Command 上设置 PATH 环境变量来控制，但这在 Windows 上有一些 实现限制 （参见 issue rust-lang/rust#37519）。

§示例

基本用法：

use tokio::process::Command;
let mut command = Command::new("sh");

pub fn arg<S: AsRef<OsStr>>(&mut self, arg: S) -> &mut Command

添加传递给程序的参数。

每次调用只能传递一个参数。因此，请勿这样：

let mut command = tokio::process::Command::new("sh");
command.arg("-C /path/to/repo");

用法如下：

let mut command = tokio::process::Command::new("sh");
command.arg("-C");
command.arg("/path/to/repo");

若要传递多个参数，请参阅 args。

§示例

基本用法：

use tokio::process::Command;

let output = Command::new("ls")
        .arg("-l")
        .arg("-a")
        .output().await.unwrap();

pub fn current_dir<P: AsRef<Path>>(&mut self, dir: P) -> &mut Command

为子进程设置工作目录。

§Platform-specific behavior

如果程序路径是相对的（例如 "./script.sh"）， 那么是应该相对于 父进程的工作目录解释，还是相对于 current_dir 解释， 这一点不明确。这种情况下的行为 是平台特定的且不稳定的， 建议使用 canonicalize 来获取绝对的程序路径。

§示例

基本用法：

use tokio::process::Command;

let output = Command::new("ls")
        .current_dir("/bin")
        .output().await.unwrap();

pub fn kill_on_drop(&mut self, kill_on_drop: bool) -> &mut Command

控制在丢弃对应的 Child 句柄时，是否应对已生成的子进程调用 kill 操作。

默认情况下，此值假定为 false，即下一个生成的进程在丢弃时不会被杀死， 与标准库的行为类似。

§Caveats

在 Unix 平台上，进程退出后必须由其父进程â回收â， 以释放所有操作系统资源。已退出但尚未被父进程回收的子进程被视为â僵尸â进程。 此类进程仍会计入系统施加的限制， 并且存在过多僵尸进程可能会阻止新进程的生成。

虽然向子进程发出 kill 信号是同步操作， 但生成的僵尸进程无法在析构函数中 .await， 以避免阻塞其他任务。Tokio 运行时将尽最大努力 在后台尝试回收和清理此类进程， 但对于此过程的发生频率和速度不提供额外保证。

如果需要更强的保证，建议尽可能避免丢弃 Child 句柄， 而应尽可能使用 child.wait().await 或 child.kill().await。

pub fn spawn(&mut self) -> Result<Child>

将命令作为子进程执行，并返回 到该进程的句柄。

默认情况下，stdin、stdout 和 stderr 继承自父进程。

此方法会同步生成子进程并返回 到 future 感知子进程的句柄。返回的 Child 自身实现 Future 以获取子进程的 ExitStatus， 此外 Child 还有 获取 stdin、stdout 和 stderr 流句柄的方法。

此子进程执行的所有 I/O 都将与 当前默认事件循环关联。

§示例

基本用法：

use tokio::process::Command;

async fn run_ls() -> std::process::ExitStatus {
    Command::new("ls")
        .spawn()
        .expect("ls command failed to start")
        .wait()
        .await
        .expect("ls command failed to run")
}
§Caveats
§Dropping/Cancellation

与标准库的行为类似，且不像 future 中"丢弃即取消"的范式，默认情况下，派生的进程即使在 Child handle 被丢弃后仍会继续执行。

可使用 Command::kill_on_drop 方法修改此行为，使得在 Child 包装器被丢弃前子进程尚未退出的情况下杀死子进程。

§Unix Processes

在 Unix 平台上，进程退出后必须由其父进程â回收â， 以释放所有操作系统资源。已退出但尚未被父进程回收的子进程被视为â僵尸â进程。 此类进程仍会计入系统施加的限制， 并且存在过多僵尸进程可能会阻止新进程的生成。

Tokio 运行时将尽最大努力 尝试回收和清理其已生成的任何进程。 对于此过程的发生频率和速度不提供额外保证。

如果需要更严格的清理保证，建议在 Child 进程 handle 被完全 await 之前避免丢弃它。

§错误

在 Unix 平台上，如果达到系统进程限制（包括系统上运行的其他应用程序）， 此方法将失败并返回 std::io::ErrorKind::WouldBlock。

pub fn status(&mut self) -> impl Future<Output = Result<ExitStatus>>

将命令作为子进程执行，等待其完成 并收集其退出状态。

默认情况下，stdin、stdout 与 stderr 继承自父进程。 如果任何输入/输出 handle 被设置为 pipe，则它们会在子进程派生后立即关闭。

此子进程执行的所有 I/O 都将与 当前默认事件循环关联。

如果 kill_on_drop 设置为 true， 此函数返回的 future 的析构函数 将杀死子进程。

§错误

如果无法生成子进程，或者 在等待其状态时发生错误，则此 future 将返回错误。

在 Unix 平台上，如果达到系统进程限制（包括系统上运行的其他应用程序）， 此方法将失败并返回 std::io::ErrorKind::WouldBlock。

§示例

基本用法：

use tokio::process::Command;

async fn run_ls() -> std::process::ExitStatus {
    Command::new("ls")
        .status()
        .await
        .expect("ls command failed to run")
}

pub fn output(&mut self) -> impl Future<Output = Result<Output>>

作为子进程执行命令，等待其完成并收集其所有输出。

注意：与标准库不同，此方法会 无条件地将 stdout/stderr 句柄配置为管道， 即使它们之前已经配置过。如果不需要这种行为， 则应将 spawn 方法与 child 上的 wait_with_output 方法结合使用。

此方法将返回一个 future，表示 收集子进程的 stdout/stderr。它将解析 为标准库中的 Output 类型，包含 stdout 和 stderr 作为 Vec<u8>，以及表示进程退出方式的 ExitStatus。

此子进程执行的所有 I/O 都将与 当前默认事件循环关联。

如果 kill_on_drop 设置为 true， 此函数返回的 future 的析构函数 将杀死子进程。

§错误

如果无法生成子进程，或者 在等待其状态时发生错误，则此 future 将返回错误。

在 Unix 平台上，如果达到系统进程限制（包括系统上运行的其他应用程序）， 此方法将失败并返回 std::io::ErrorKind::WouldBlock。

§示例

基本用法：

use tokio::process::Command;

async fn run_ls() {
    let output: std::process::Output = Command::new("ls")
        .output()
        .await
        .expect("ls command failed to run");
    println!("stderr of ls: {:?}", output.stderr);
}

pub fn get_kill_on_drop(&self) -> bool

返回之前由 Command::kill_on_drop 设置的布尔值。

请注意，如果你之前没有调用 Command::kill_on_drop， 则此处将返回默认值 false。

§示例
use tokio::process::Command;

let mut cmd = Command::new("echo");
assert!(!cmd.get_kill_on_drop());

cmd.kill_on_drop(true);
assert!(cmd.get_kill_on_drop());
