宏 select 

Source 
macro_rules! select {
    {
        $(
            biased;
        )?
        $(
            $bind:pat = $fut:expr $(, if $cond:expr)? => $handler:expr,
        )*
        $(
            else => $els:expr $(,)?
        )?
    } => { ... };
}
展开描述

等待多个并发分支，当第一个分支完成时返回， 并取消其余分支。

select! 宏必须在 async 函数、闭包和块内使用。

select! 宏接受一个或多个具有以下模式的分支：

<pattern> = <async expression> (, if <precondition>)? => <handler>,

此外，select! 宏可以包含一个可选的 else 分支， 在没有其他分支匹配其模式时进行求值：

else => <expression>

宏会聚合所有 <async expression> 表达式， 并在当前任务上并发运行它们。一旦第一个表达式 以匹配其 <pattern> 的值完成，select! 宏 就会返回已完成分支的 <handler> 表达式的求值结果。

此外，每个分支可以包含一个可选的 if 前置条件。如果 前置条件返回 false，则该分支被禁用。提供的 <async expression> 仍会被求值，但生成的 future 永远不会被 轮询。此功能在循环中使用 select! 时非常有用。

select! 表达式的完整生命周期如下：

  1. Evaluate all provided <precondition> expressions. If the precondition returns false, disable the branch for the remainder of the current call to select!. Re-entering select! due to a loop clears the “disabled” state.
  2. Aggregate the <async expression>s from each branch, including the disabled ones. If the branch is disabled, <async expression> is still evaluated, but the resulting future is not polled.
  3. If all branches are disabled: go to step 6.
  4. Concurrently await on the results for all remaining <async expression>s.
  5. Once an <async expression> returns a value, attempt to apply the value to the provided <pattern>. If the pattern matches, evaluate the <handler> and return. If the pattern does not match, disable the current branch for the remainder of the current call to select!. Continue from step 3.
  6. Evaluate the else expression. If no else expression is provided, panic.

§Runtime characteristics

由于所有异步表达式都在当前任务上运行，这些表达式 能够并发运行，但不能并行运行。这意味着所有 表达式都在同一线程上运行，如果某个分支阻塞了线程， 所有其他表达式都将无法继续。如果需要并行执行， 请使用 tokio::spawn 生成每个异步表达式， 然后将其 join handle 传递给 select!。

§Fairness

默认情况下，select! 会随机选择一个分支优先检查。 当在循环中调用 select! 且分支始终就绪时， 这提供了一定程度的公平性。

可以通过在宏使用的开头添加 biased; 来覆盖此行为。 有关详细信息，请参阅示例。这将使 select 按 future 出现的顺序（从上到下）进行轮询。出于以下几种原因，你可能希望这样做：

  • The random number generation of tokio::select! has a non-zero CPU cost
  • Your futures may interact in a way where known polling order is significant

但此模式有一个重要的注意事项。确保你的 future 的轮询顺序是公平的 成为了你的责任。例如，如果你在流和关闭 future 之间进行选择， 而流有大量消息且消息之间的时间间隔为零或几乎为零， 则应将关闭 future 放在 select! 列表的更靠前位置， 以确保它始终被轮询，并且不会因为流始终就绪而被忽略。

§Panics

如果所有分支都被禁用且没有提供 else 分支， 则 select! 宏会 panic。当提供的 if 前置条件返回 false 或模式与 <async expression> 的结果不匹配时，分支被禁用。

§Cancellation safety

在循环中使用 select! 从多个来源接收消息时， 应确保接收调用是可取消安全的，以避免 丢失消息。本节介绍各种常见方法并 说明它们是否是可取消安全的。本节中的列表并不 详尽。

以下方法是可取消安全的：

  • tokio::sync::mpsc::Receiver::recv
  • tokio::sync::mpsc::UnboundedReceiver::recv
  • tokio::sync::broadcast::Receiver::recv
  • tokio::sync::watch::Receiver::changed
  • tokio::net::TcpListener::accept
  • tokio::net::UnixListener::accept
  • tokio::signal::unix::Signal::recv
  • tokio::io::AsyncReadExt::read on any AsyncRead
  • tokio::io::AsyncReadExt::read_buf on any AsyncRead
  • tokio::io::AsyncWriteExt::write on any AsyncWrite
  • tokio::io::AsyncWriteExt::write_buf on any AsyncWrite
  • tokio_stream::StreamExt::next on any Stream
  • futures::stream::StreamExt::next on any Stream

以下方法不是可取消安全的，可能导致数据丢失：

  • tokio::io::AsyncReadExt::read_exact
  • tokio::io::AsyncReadExt::read_to_end
  • tokio::io::AsyncReadExt::read_to_string
  • tokio::io::AsyncWriteExt::write_all

以下方法不是可取消安全的，因为它们使用队列以保证公平性， 而取消会使你在队列中失去位置：

  • tokio::sync::Mutex::lock
  • tokio::sync::RwLock::read
  • tokio::sync::RwLock::write
  • tokio::sync::Semaphore::acquire
  • tokio::sync::Notify::notified

要判断你自己的方法是否可取消安全，请查找 .await 的使用位置。这是因为当异步方法被 取消时，这总是发生在 .await 处。如果你的函数 即使在 .await 处等待时被重启也能正确运行， 那么它就是可取消安全的。

可以按以下方式定义可取消安全：如果你有一个 尚未完成的 future，那么丢弃该 future 并重新创建它必须是无操作（no-op）。此定义源自于 在循环中使用 select! 的场景。如果没有此保证， 当另一个分支完成时，你将失去进度， 并通过循环重新启动 select!。

请注意，取消不可取消安全的操作 并不一定是错误的。例如，如果你因为 应用程序关闭而取消任务，那么你可能不 关心部分读取的数据丢失。

§示例

具有两个分支的基础 select。

async fn do_stuff_async() {
    // async work
}

async fn more_async_work() {
    // more here
}

tokio::select! {
    _ = do_stuff_async() => {
        println!("do_stuff_async() completed first")
    }
    _ = more_async_work() => {
        println!("more_async_work() completed first")
    }
};

基础流选择。

use tokio_stream::{self as stream, StreamExt};

let mut stream1 = stream::iter(vec![1, 2, 3]);
let mut stream2 = stream::iter(vec![4, 5, 6]);

let next = tokio::select! {
    v = stream1.next() => v.unwrap(),
    v = stream2.next() => v.unwrap(),
};

assert!(next == 1 || next == 4);

收集两个流的内容。在本例中，我们依赖模式 匹配以及 stream::iter 是â融合（fused）â的这一事实， 即一旦流完成，对 next() 的所有调用都返回 None。

use tokio_stream::{self as stream, StreamExt};

let mut stream1 = stream::iter(vec![1, 2, 3]);
let mut stream2 = stream::iter(vec![4, 5, 6]);

let mut values = vec![];

loop {
    tokio::select! {
        Some(v) = stream1.next() => values.push(v),
        Some(v) = stream2.next() => values.push(v),
        else => break,
    }
}

values.sort();
assert_eq!(&[1, 2, 3, 4, 5, 6], &values[..]);

在多个 select! 表达式中使用同一个 future 可以通过传递 该 future 的引用来实现。这要求该 future 必须是 Unpin 的。 可以通过使用 Box::pin 或栈固定（stack pinning） 将 future 变为 Unpin。

在此示例中，一个流最多被消耗 1 秒。

use tokio_stream::{self as stream, StreamExt};
use tokio::time::{self, Duration};

let mut stream = stream::iter(vec![1, 2, 3]);
let sleep = time::sleep(Duration::from_secs(1));
tokio::pin!(sleep);

loop {
    tokio::select! {
        maybe_v = stream.next() => {
            if let Some(v) = maybe_v {
                println!("got = {}", v);
            } else {
                break;
            }
        }
        _ = &mut sleep => {
            println!("timeout");
            break;
        }
    }
}

使用 select! 连接两个值。

use tokio::sync::oneshot;

let (tx1, mut rx1) = oneshot::channel();
let (tx2, mut rx2) = oneshot::channel();

tokio::spawn(async move {
    tx1.send("first").unwrap();
});

tokio::spawn(async move {
    tx2.send("second").unwrap();
});

let mut a = None;
let mut b = None;

while a.is_none() || b.is_none() {
    tokio::select! {
        v1 = (&mut rx1), if a.is_none() => a = Some(v1.unwrap()),
        v2 = (&mut rx2), if b.is_none() => b = Some(v2.unwrap()),
    }
}

let res = (a.unwrap(), b.unwrap());

assert_eq!(res.0, "first");
assert_eq!(res.1, "second");

使用 biased; 模式控制轮询顺序。

let mut count = 0u8;

loop {
    tokio::select! {
        // If you run this example without `biased;`, the polling order is
        // pseudo-random, and the assertions on the value of count will
        // (probably) fail.
        biased;

        _ = async {}, if count < 1 => {
            count += 1;
            assert_eq!(count, 1);
        }
        _ = async {}, if count < 2 => {
            count += 1;
            assert_eq!(count, 2);
        }
        _ = async {}, if count < 3 => {
            count += 1;
            assert_eq!(count, 3);
        }
        _ = async {}, if count < 4 => {
            count += 1;
            assert_eq!(count, 4);
        }

        else => {
            break;
        }
    };
}

§Avoid racy if preconditions

由于 if 前置条件用于禁用 select! 分支， 因此必须谨慎使用以避免遗漏值。

例如，下面是 sleep 与 if 的错误用法。目标是 重复运行异步任务最多 50 毫秒。 但是，sleep 的完成有被遗漏的可能。

ⓘ
use tokio::time::{self, Duration};

async fn some_async_work() {
    // do work
}

let sleep = time::sleep(Duration::from_millis(50));
tokio::pin!(sleep);

while !sleep.is_elapsed() {
    tokio::select! {
        _ = &mut sleep, if !sleep.is_elapsed() => {
            println!("operation timed out");
        }
        _ = some_async_work() => {
            println!("operation completed");
        }
    }
}

panic!("This example shows how not to do it!");

在上面的示例中，即使 sleep.poll() 始终未返回 Ready， sleep.is_elapsed() 也可能返回 true。这会造成潜在的竞态条件—— sleep 在 while !sleep.is_elapsed() 检查与 对 select! 的调用之间到期，导致 some_async_work() 调用 在 sleep 已到期的情况下仍不间断地运行。

在不存在竞态条件的情况下编写上述示例的一种方法是：

use tokio::time::{self, Duration};

async fn some_async_work() {
    // do work
}

let sleep = time::sleep(Duration::from_millis(50));
tokio::pin!(sleep);

loop {
    tokio::select! {
        _ = &mut sleep => {
            println!("operation timed out");
            break;
        }
        _ = some_async_work() => {
            println!("operation completed");
        }
    }
}

§Alternatives from the Ecosystem

select! 宏是管理多个异步分支的强大工具， 使得任务能够在同一线程内并发运行。然而， 其使用也可能带来挑战，特别是关于可取消安全， 这可能导致微妙且难以调试的错误。对于许多用例， 生态系统的替代方案可能更可取，因为它们通过提供 更清晰的语法、更可预测的控制流来缓解这些问题， 并减少手动处理熔断语义或可取消安全等问题的需要。

§Merging Streams

对于使用 loop { select! { ... } } 来轮询多个任务的情况， 流合并提供了一种简洁的替代方案，天生支持可取消安全的 处理，消除了数据丢失的风险。诸如 tokio_stream、 futures::stream 和 futures_concurrency 等库提供了用于合并 流并按顺序处理其输出的工具。

§Example with select!

struct File;
struct Channel;
struct Socket;

impl Socket {
    async fn read_packet(&mut self) -> Vec<u8> {
        vec![]
    }
}

async fn read_send(_file: &mut File, _channel: &mut Channel) {
    // do work that is not cancel safe
}

// open our IO types
let mut file = File;
let mut channel = Channel;
let mut socket = Socket;

loop {
    tokio::select! {
        _ = read_send(&mut file, &mut channel) => { /* ... */ },
        _data = socket.read_packet() => { /* ... */ }
        _ = futures::future::ready(()) => break
    }
}

§Moving to merge

通过使用 merge，可以将多个异步任务统一到单个流中， 无需手动管理任务，并降低意外行为（如数据丢失）的风险。

use std::pin::pin;

use futures::stream::unfold;
use tokio_stream::StreamExt;

struct File;
struct Channel;
struct Socket;

impl Socket {
    async fn read_packet(&mut self) -> Vec<u8> {
        vec![]
    }
}

async fn read_send(_file: &mut File, _channel: &mut Channel) {
    // do work that is not cancel safe
}

enum Message {
    Stop,
    Sent,
    Data(Vec<u8>),
}

// open our IO types
let file = File;
let channel = Channel;
let socket = Socket;

let a = unfold((file, channel), |(mut file, mut channel)| async {
    read_send(&mut file, &mut channel).await;
    Some((Message::Sent, (file, channel)))
});
let b = unfold(socket, |mut socket| async {
    let data = socket.read_packet().await;
    Some((Message::Data(data), socket))
});
let c = tokio_stream::iter([Message::Stop]);

let mut s = pin!(a.merge(b).merge(c));
while let Some(msg) = s.next().await {
    match msg {
        Message::Data(_data) => { /* ... */ }
        Message::Sent => continue,
        Message::Stop => break,
    }
}

§Racing Futures

如果你需要等待几个异步任务中的第一个完成， 可以使用以下生态系统实用工具： futures、 futures-lite 或 futures-concurrency， 它们提供了用于 race future 的简洁语法：

  • futures_concurrency::future::Race
  • futures::select
  • futures::stream::select_all (for streams)
  • futures_lite::future::or
  • futures_lite::future::race
use futures_concurrency::future::Race;

let task_a = async { Ok("ok") };
let task_b = async { Err("error") };
let result = (task_a, task_b).race().await;

match result {
    Ok(output) => println!("First task completed with: {output}"),
